Benefits Exemplified

This section gives details and examples of the Benefits of using the Tymeac System found in the Product description.

Control for Asynchronous Threads

See Product description, Control for Asynchronous Threads.

Multi-threading goes back to the 1950's. Even languages like FORTRAN support multi-threading.

What they do not support is a way to manage the threads, see problems. 

An example of unmanaged threads is the Demonstration System multiple event Class. 

TyDemoT3 is the Demonstration Class that can put a load on your system.

TyDemoT3Bean, TyDemoT3Base, TyDemoT3Instant and TyDemoT3Thread are Classes in support thereof. 

TyDemoT3 does not manage the synchronous threads (TyDemoT3Thread). If something goes wrong, so what. Just kill the process and start over. This is simply a demonstration. We provide the source code. You can add/alter that which you like. Make the base array volatile. Add synchronized methods. This is not mission critical.

Tymeac can handle your mission critical applications.

When applications run on servers in remote locations there is no easy way to kill the process and start over.  When applications need multi phase commits across the network or handle money, they must be secure, reliable, manageable and fault tolerant.

Tymeac is secure.

Tymeac runs on each server without direct connections to any other server [single affinity].

Tymeac has no communication facilities. All RMI and ClassLoader processing within Tymeac is for local resources, only (TyVariables, url_name).  Processing Application Classes load with a custom loader for the local system only (findSystemClass()). Even new versions of Processing Application Classes use a custom loader for the local system only (and a new Name Space).

This ensures that no unauthorized modules enter the system.

While it is true that your Processing Application Classes may violate encapsulation, Tymeac itself is secure.

Tymeac is reliable.

Tymeac™ was originally designed to manage asynchronous tasks in the transaction processing environment.

This is the 24/7 arena of international credit authorization, money transfers and all mission-critical applications.

Tymeac™ grew up with the principal of "no excuses". Today, all the features that make Tymeac™ a highly reliable, fully mission-critical software product are available for the Java™ Virtual Machine.

  • total number of 'threads per Queue' limits to prevent thread overload,
  • thread creation / destruction as necessary,
  • thread activation according to thresholds, not chance,
  • prioritized Queues for requests,
  • extensive monitoring of the thread environment to pinpoint stalls,
  • integrated re-scheduling for stalled requests,
  • interfacing with user-controlled notification systems when problems arise,
  • support for synchronous (timed) requests as well as asynchronous (background) requests,
  • GUI support to monitor and alter the environment for system personnel,
  • logging of statistics at shutdown and on request,
  • support for any DBMS through JDBC, to optionally store parameters and logs,
  • support for any embedded DBMS,
  • support for non-DBMS parameters for a minimal footprint.

Tymeac is manageable.

Tymeac's Functions and Queues were designed for manageability.

Adding or removing a Queue (Normal or Output Agent) from a Function is only a matter of a table change.

Changing the Processing Application Class that executes within a thread is only a matter of a table change.

Adding or removing threads from a Queue is only a matter of a table change.

Tymeac provides almost unlimited control for managing the new thread algorithm. This assures that only those threads that are immediately necessary to process requests are active, saving valuable processor resources. 

Prioritized Wait Lists provide the means to properly schedule requests and to assure users that the hot request gets immediate attention.

The entire execution environment is viewable (Threads, Wait Lists). Thread status is alterable. Queue data including the number of  Wait List entries is alterable.

System statistics log at shut down and on request.

Tymeac is fault tolerant.

Optional timing of user requests provides immediate notification of an abnormal event.

Automatic back out provides for intermediate component failure.

A Monitor thread thoroughly examines the system at set intervals reallocating and freeing resources, and, notifying the user’s early warning system of potential problems.

Tymeac is the easiest way to run a background process. The background process, or, back end -- back room, is common in the Enterprise. The Tymeac Asynchronous Request method is the means to feed the background task. The Tymeac Client accepts data and passes the Object to the Tymeac Server. The background process executes without any intervention from the requesting Tymeac Client. The examples, revalue portfolio, is also an example of background processing.

Reduced Processor Overhead

See Product description, Reduced Processor Overhead.

The Messaging API and the DataBase API (JDBC) provide two prime examples of how Tymeac can cut overhead in your processing environment.

Set up a Center (Messaging, DataBase, etc). Access to the Center is through a transport mechanism (RMI and CORBA are two such transports). When the first request arrives for a particular service, open the service and place the request therein.  As each new request arrives, place the request therein. When at a threshold (either request limit or time-out) close the service.

In this way, multiple requests can process between the open and the close.

See the Demonstration System. The Tymeac Functions for asynchronous processing with an Output Agent Queue are Functions 5, 6, and 7. The Output Agent Processing Application Class for these Tymeac Functions is DemoAgent1. Alter this Class to send the results of local processing to your Center.

Scalability

See Product description, Scalability

How many threads does a logical process need? The answer depends on the number of outstanding requests (load). The higher the load, the more threads may be needed to process requests.

Tymeac™ provides four thresholds to give designers control over this parameter. Tymeac™ creates a new thread either at start up or when first needed (threshold processing) and destroys the thread either after an idle period or never. This is a dynamic response to an ever changing environment.

Facilitated Recursion Processing

See Product description, Nested Processing

See pattern, Recursion.

Recursion is taught in compiler theory courses. The standard example is the parenthesis processor. Most languages support recursion, even going back as far as PL/1.

What is not standard is how a separate, executing "process", can call itself or another as a new thread. Just keeping track of the levels is exhausting. Add in component failures and stalls, and you need Tymeac.

The Demonstration System contains an example of recursion. Function_8 is the recursive processing Function. The examples, revalue portfolio, is also an example of how to use Tymeac recursion.

The Notification System is a recursive example. The Tymeac Monitor, when it finds a problem, calls itself, the Tymeac Server, as a Tymeac Client.

Simplified Application Development

See Product description, Concurrent Development.

Tymeac supports the construction of simple, single process objects. These Processing Application Classes run under Tymeac Threads. The object gets the input message from Tymeac, executes the process, and returns the output message back to Tymeac.

Tymeac allows object development to proceed in a concurrent manor. Tymeac allows incremental prototyping and development of all tasks simultaneously. The design, construction, and testing of objects in a vacuum simplifies the entire process. Redesign of the object does not effect other objects and can proceed without shutting down the system. Since Tymeac allows objects to execute as separate threads, the vacuum development environment reflects the real world environment.

Once written and tested, these objects then become part of your repository. Since they are Java™ objects, and Tymeac itself is a 100% pure application, they can run on any hardware, anywhere. See design pattern, Use-Sharing.

See the Demonstration System. Each Processing Application Class is small, single purpose. Getting a final result is simply a matter of stringing together the Tymeac Queues into a Tymeac Function.

Reuse has become a reality.

Failure Containment

See Product description, Failure Containment

Failures propagate up the chain. They also propagate sideways, downward, halfway in between and around. Anyone who writes programs outside the stand-alone world knows this.

Other industries have means to limit exposure from the harmful:

The nuclear industry has containment vessels to hold the radio active material.

The biological industry has sealed laboratories to contain the deadly germs.

Until the day comes that computer programs can be 100% thoroughly tested in all situations, then a means for isolating code is desirable.

Tymeac provides the means to isolate the code. Each Processing Application Class is separate from the other Processing Application Classes (see pattern, Mediator). Each Processing Application Class can fail without affecting any other Processing Application Class. Tymeac automatically backs out intermediate component failures when Queues fail to schedule. Tymeac testing is easier than the traditional approach.

When your Processing Application Class is a front end for Messaging, Data Base, or transport mechanisms, then the isolation environment of Tymeac can save your business immeasurable hours of debugging. There is no need to sift through extraneous elements of the matrix. See also Gateway, below.

Separately Tunable JVM's

See Product description, Tunable JVM's.

Tymeac Server runs in its own Java Virtual Machine, separate from your application Client, Data Base Management System, Jini Service or other middle tier layer. This advantage lets you:

  • Use separate security property and/or classpath options. You can have the same process resident in several JVM's each with its own -Dsecurity.policy.file or -classpath.  In this way, you can control who does what, where, and when.

  • Personnel may quickly move the JVM to the most advantageous machine when needed. This is as easy as changing a script file and moving it to another machine along with the Tymeac class files.

  • Personnel may add a debugging interface or profiling option. Since each JVM is separate, stopping the processing of one does not impact any other.

  • Personnel may change memory allocation or other non-standard start up options. Non-standard options are good for debugging and tuning. However, Sun may not support each indefinitely or in the same way. Using a separate JVM gives you the option of using what is best now.

Gateway to Distributed Computing

See Product description, Gateway.

Tymeac uses an n-tiered architecture. The Tymeac process is separate from the database and the user interface.

Each Tymeac component process executes as a separate thread. The communication, application and database interface dependencies normally found in large, multi-function processes do not exist. See pattern, Adapter.

As Java™ expands into transaction processing, the Logical Unit of Work (LUW) becomes a factor. A complex process (Tymeac Function) can have multiple logical units of work (threads in Tymeac Queues). Each LUW may interface with [including the Java Native Interface] and hold locks on, a different resource.

Physically distributing the work is the key to Distributed Computing. Once the work is in separate threads, each independent thread may use any method for transport (RMI, CORBA, etc.) to another location. Each thread is in isolation from the other threads. This also simplifies testing and debugging, see Failure Containment, above.

The examples, revalue portfolio, is also an example of how to distribute the work.

Ability to Multi-Process

See Product description, Multi Processing.

Multi Processing is the ability to put different threads on different CPU's within the same box. This does not happen by magic. Even FORTRAN must segment the program explicitly.

Separating the work into threads is mandatory. Managing those threads is Tymeac. See Control, above.

Trying to manage multiple processes in an MPP or SMP environment is difficult. Tymeac manages multiple processes in unique Queues. The developer codes the method without concern for the execution environment.

Tymeac lets the operating system, Java™, and access methods schedule, monitor and control the use of resources by separating complex requests into component threads.

See the Demonstration System. Any Tymeac Queue can run on any CPU and produce the proper result.
