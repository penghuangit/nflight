Installation Verification Procedure

The purpose of this section is to verify that Tymeac installed fully. Total functionality, beyond verification, requires using the Demonstration System and altering the sample application classes we supply with the product.

Preliminary:

The product ships for the SDK1.6.+ environment. If you use a prior version, then you should download a prior version of Tymeac. (version 5.5 supported JDK1.5)

We supply script files for the major functions of the product. It is much easier to run a script than to enter a series of characters on the command line. Although these scripts are optional, it may be beneficial to edit those scripts here rather than as they become necessary.

The scripts, (both .bat and .sh), follow the directory structure mentioned previously. At the minimum, you must edit the SET_ENV script to put in where you installed the Tymeac Classes. You should do this NOW: See the scripts document.

The two scripts necessary for this IVP (and two others that are necessary shortly thereafter) are:

  • RMIServer  --  Starts the Tymeac Server.

  • T1  --  Tymeac Client access for a single request

  • T3  --   Tymeac Client access for a multiple requests

  • menu   --  Starts the menu of Tymeac frame classes

The Registry:

This version of Tymeac is an RMI Server. Therefore, the RMI Registry must be running.

If you do not have a network, then you must use the internal server. You may wish to use the internal server just to verify that the classes are in order. Then return here for RMI.

Running the internal server.

Your CLASSPATH must have at least the <home> directory so that the RMI Registry may find the stub class for marshaling. Alternatively, you may start the RMI Registry from the <home> directory (this is then the current directory for the Registry.)

The CLASSPATH for the registry is the single most reason for RMI failure. If you start the Server and get a
    java.rmiServerException
    java.rmiUnmarshalException
    java.lang.ClassNotFoundException com.tymeac.base.TymeacStandardImpl_Stub
Then you have a CLASSPATH error.

After you verify installation, you may use code base or any other method for finding remote objects.

Starting the RMI Registry is different for each operating system. The terminology is different for each operating system, but it requires starting the registry as a separate "process", "application", "window", "task", etc. We mention several operating systems here:

Windows™ : start rmiregistry
Unix® : rmiregistry &

Using "frames", "windows", "screens" is different in each operating environment. In order to use the frame classes in Tymeac your environment must be "window oriented". On some systems, such as a Unix® variant, you must enable the window environment with a command such as: startx.

Tymeac Server writes messages to the console. This is System.out.println(). (After Verification you can suppress this on the Command Line Start-up.) Therefore, enabling a console or Standard Out is helpful.

For this discussion, we use the Windows format of a back slash, (\).

Tymeac Server:

Start Tymeac, Java Class is com.tymeac.base.RMIServer.

We supply a script to start the TymeacServer: RMIServer.bat[sh].

Tymeac Message structure is here. 
Tymeac Server Start Up messages are detailed here.
The one you need is that the start up is successful. 

Ty00099I Tymeac Server initialized successfully, MilliTime=1168010322254

MilliTime is the System time in milliseconds since 1970. (The words MilliTime and Session are synonymous for Tymeac.) 
MilliTime appears on the successful start up message, the shut down message, and within all return messages for Asynchronous Requests.
It is the way to identify this session of Tymeac.

Missing Classes or "something not found" usually means that the path to Tymeac Classes is not proper or a spelling or case sensitive error.

For verification, Tymeac uses no logging or statistics.

Note:  Tymeac starts various threads during start up to perform necessary functions. Each of these threads writes its own messages. Therefore, the order of these messages is unpredictable.

Tymeac Client:

Tymeac Client access is in Frame Class, TyDemoT1. You may refer to the demonstration document.

We supply a script file for this, T1.bat [sh].

  Fill in a Case Sensitive Function Name: "Function_2". [We supply Function_1 through Function_8]. 

The F in Function is a Capital F
Since this is a String, there must not be any trailing spaces.

Press the <SEND> button.

  The result should be: Tymeac SR(0000)#Demo1:A1(00)+A2(00)+A3(00)=(0)#

  • Tymeac is the prefix.
  • SR stands for Synchronous Request.
  • (nnnn) Return code, should be 0000.  When the number in parentheses is not zero, then check the return code document for the failure reason.
  • # ... # the message within the pound signs is dependent on the values in the A, B, and C TextFields.  For details, see the TyDemoT1 document .

  As long as you get a return message, the installation is successful.

Communication errors: If you have connection problems, then you may alter the catch (Exception e) code in Methods in the Tymeac Server Communication Class, TySvrComm, by removing the commenting on the print statements.

Shut down:

The graceful termination of Tymeac Server is possible. RMI Servers start a thread that never ends. (The Tymeac Server also issues a never ending wait() so that necessary variables do not get garbage collected.) This is so that the Server is persistent until manually shut down. Tymeac starts a termination thread during shut down, when all threads are complete. The thread calls System.exit(0) which terminates the Java Virtual Machine.

Tymeac shuts down in two stages. The first informs any live Queue Threads to complete and terminate. The second forces termination of any remaining Queue Threads and starts the termination thread. Therefore, you may have to request shut down at least twice (if Queue Threads were alive) and sometimes three times. (If necessary, see the discussion of Output Agent Queue Threads in the shut down section.)

Two methods for shutting down Tymeac Server are available. A Frame Class (from the main menu) and a command line Class.

Frame:  Menu

We supply a script file for this, Menu.bat [sh].

Select the Shut Down button.

Press the SHUT Button.
    Observer the message and if necessary
Press the SHUT Button.
     Observer the message.

Command Line:  Using the script, shut.

Do this twice, if necessary.

The ungraceful termination of Tymeac Server is possible by just killing the Tymeac Server process.

As of this writing, the authors know of no non-programming way to gracefully shut down the RMI Registry.
