RMI Background

It is not the purpose of this document to substitute for an RMI primer.  If you need detailed information on how RMI works, then we strongly suggest you get one of the numerous books on the subject.

In order to use the Tymeac product, you only need a rudimentary knowledge of RMI.  Sun provides this in the JDK documentation:

General Information
    Guide to Features
        Basic Features
            Remote Method Invocation (RMI)
                RMI Tutorial
                    Getting started

Remote Method Invocation is similar to Remote Procedure Call. A client, here, calls a method on a server, there.  For RPC the word "Remote" means "Not in this computer". For RMI the word "Remote" simply means: "Not in this Java Virtual Machine". For RMI, you may run the client and the server on the same machine, but in different JVM's.

An RMI client naturally must know what methods are available on the server JVM. In RMI, a Java interface describes these methods. The RMI server implements the interface methods, not the RMI clients. Since the RMI clients do not have direct access to the implementation of the RMI server methods, the RMI compiler must produce an object for use by the RMI clients. This object is called the stub. For marshalling on the server in JDK1.1, RMI uses a similar object called the skel.

In order for clients to find servers, RMI uses a process called the bootstrap registry service or the Java Remote Object Registry or simply the RMI Registry. This native executable is in the "bin" directory as part of the Java Runtime Library on you PATH.

For CORBA, there is the CosNaming plugin for the JNDI Registry. If you wish to use this feature, then see:

Guide to Features
    Enterprise Features
        RMI-IIOP

The RMI server registers objects (the implementation of the interface) with the RMI Registry by passing a reference to the object and giving the object a java.lang.String "name".

The clients retrieve stubs to those objects by specifying the same java.lang.String "name" and communicating with the RMI Registry through a URL that begins with "rmi://".  When the client and server are on on the same machine, the URL is "rmi:///". When on different machines, the URL points to that machine, "rmi://your-company.com/".

Note the three slashes (///) for a local URL. This is not a typo. That third slash is a trailing slash.  In order to set this to a remote name, you put that remote name between the second and third slashes.  Such as:
rmi://your-company.org/  or  rmi://192.168.42.1/  (using a physical IP address is handy for testing.)

The only concern now is how does the RMI Registry find the Interface.class, stub. The answer is the CLASSPATH. When the RMI server registers with the RMI Registry, the Registry needs access to these objects.

RMI clients, when on a separate machine, MAY not need a CLASSPATH. This is dependent on your operating system and Java release.

The interface (TymeacInterface.class),  stub ( TymeacStandardImpl_Stub) and (TymeacActivationImpl_Stubl for the activation system), reside in the Tymeac directory: <home>\base. (See the directory document for a description of all the Tymeac directories.)

Even if you use the -classpath option, the RMI Registry still needs access to these objects and this is through the CLASSPATH.

An unprofessional approach to bypass using a CLASSPATH is to switch directories to the <home>\base directory and start the RMI Registry therefrom. Since command windows inherit the ClassPath of the command window in which they start, the RMI Registry finds the stub by default.

If you do not provide proper CLASSPATH access to these objects, then you will get unMarshaling errors on Tymeac start up and/or on Tymeac client communication.
