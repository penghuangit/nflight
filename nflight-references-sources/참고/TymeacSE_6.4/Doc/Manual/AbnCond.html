Abnormal Conditions

In this section we discuss the abnormal conditions that arise when using the Tymeac system.

Abnormal Conditions

How to force a timeout:

The supplied Tymeac client classes call the Tymeac Server, for a Synchronous Request, passing a parameter, TymeacParm.  The third field is the maximum time to wait for all Queues to finish processing, after all Queues schedule.  A timeout is when all Queues do not finish within this time.  Therefore, to force a timeout, delay the processing of one or more of the Processing Application Classes.

A suggestion:  For any Function, Queue "AAAA", (all Functions in our samples use Queue "AAAA"), that uses Processing Application Class, Demo1: Add a delay to the code at the beginning of the main() method, such as:

try {
    Thread.sleep(40000)
}
catch (InterruptedException e) {}

The 40,000 is milliseconds. Therefore, it is forty (40) seconds.  None of our supplied Tymeac Client Classes wait more than thirty (30) seconds so this forces a timeout.  Naturally, if you alter any of the Tymeac Client Classes than you must adjust the sleep time accordingly.

How to force a stall:

Asynchronous requests may stall for a variety of reasons. The most common reason is that the processing is taking too long.

In order to force this type of condition, use the example of the timeout we mention, above.  Only increase the sleep time to several minutes, (i.e (60000 * 3) is three minutes.) The sleep time must be greater than the Monitor timing of the time the Queue Thread invokes the Processing Application Class.

A more difficult way to force this type of condition is to use the priority of the request.  Tymeac places a request into the Queue's Wait Lists according to its priority, (fourth parameter in TymeacParm.)  The TyDemoT3, frame sample, ships with the priority set at 1.  This sample places a "load" on your system. All requests go into the first Wait List on each Queue.

Alter either one or more of the command line Classes, (TyDemoClient_5 -7) or the TyDemoT1, frame Class to use a priority of 7. All requests go into the seventh Wait List on each Queue.

Start the TyDemoT3 Class. Start a sufficient number of threads to put a heavy load on your system.

Run one of the command line Classes or the T1 Class that you altered, above. The TyDemoT3 threads put requests into the first Wait List.  These process before the requests in the seventh Wait List. Therefore, when the load is heavy enough, those requests in the seventh Wait List do not get processing time and stall.

Another reason is an abnormal condition in a Processing Application Class.

Another way to force a stall is to abnormally terminate the Processing Application Class, Demo1-3. Add an exception to the code at the beginning of the main() method, such as:

int error = 0;
int set = 25 / error;

Or throw an exception.
