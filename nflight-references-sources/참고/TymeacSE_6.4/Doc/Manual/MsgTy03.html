Messages: Monitor Ty03

This section details the Tymeac Monitor messages.

(P) Printed to System.out.println(), (this is optional).
(L) Sent to the Tymeac Log.
(N) Sent to the Tymeac Notification Function.

001

Thread disabled reason (Ty03001)

The thread used excessive time (see how a thread gets disabled and the timing thereof.) The thread was cancelled. On the next run of the Monitor, the Monitor sets this thread to Disabled with this status code. See also 019 below.

 

005 C (N) (L)

Stalled asynchronous request are pending action.

On the tenth cycle and every tenth cycle thereafter, when the monitor finds a stalled request, it writes this message.  See how a request becomes stalled.  Purge the requests if needed.

013 I (N) (P) (L)

The monitor interval is xxx[, with inactivity deactivation in nnn minutes.]

The Monitor interval is xxx seconds. When using the activation system, deactivation will occur in nnn minutes.

015 I (L)

New stall table entry: xx

When the Monitor finds an asynchronous request that it believes is stalled, it logs this message.  xx is the Request Id

019 I (N)(L)

Tymeac Server a thread is disabled in Queue (xxxx), Disabled by reason yyyy, from cancelled reason zzzz

When the Monitor finds a cancelled thread, it sets that thread to disabled. xxxx is the Queue.

yyyy is the reason for the disable [this is usually Ty03001, see above] but it could be found here.

zzzz is the previous cancel reason. This has to do with the thread using excessive time (see how a thread gets disabled and the timing thereof.)

EXCESSIVE_PROCESSING: In a thread processing state for excessive time.

EXCESSIVE_APPL_PROCESSING_DEFAULT: In the application class state for excessive time when no override in time-out in the Queue.

EXCESSIVE_APPL_PROCESSING_SPECIFIC: In the application class state for excessive time when using the override time-out in the Queue.

EXCESSIVE_SCHD_INTERVAL: The Queue Thread is scheduling the Output Agent. In this state for excessive time.

EXCESSIVE_POSTING_INTERVAL: The thread wait waiting for work and was awakened. In this state for excessive time.

EXCESSIVE_STARTED_INTERVAL: No thread was attached. A new thread was started but had not begun execution. In this state for excessive time.

020 I (N) (L)

All threads are disabled in Queue [name]

The Queue in not capable of processing any requests.  When the Monitor first encounters this problem it writes this message.

When the Processing Application Class is faulty:

  • You can repair and compile the code.  Shut down the Tymeac Server.  Start up the Tymeac Server.
  • You can repair, rename the class and compile the new code.  Without shutting down the Tymeac Server you may use the New Copy Class, TyNewCopy to load the new Class for the Queue.  Use the Queue Thread Class, TyQueThd to enable the threads.

025 I (N) (L)

All threads are disabled in Queue [name]

This is identical to 020, above. However, the problem persists for more than a cycle of the Monitor.

098 I  (L)

Tymeac Server, unregister complete for MilliTime= xx

The Server is shut down by a shut down request and unregistered from the Activation System.  xx is the Tymeac Server start up time in milliseconds as found on the start up message, Ty00001I.

099 I  (L)

Tymeac Server, inactivation complete for MilliTime= xx

The Server is shut down by deactivation, not a shut down request.  xx is the Tymeac Server start up time in milliseconds as found on the start up message, Ty00001I.

100 I  (L)

The String message for an UnknownObjectException

No activity transpired for the requested period or a shutdown was requested.  The monitor issued the method: java.rmi.activation.Activatable.inactive() / or unregister(). RMI  threw an UnknownObjectException.

The server exits with a System.exit().

105 I  (L)

The String message for a RemoteException

No activity transpired for the requested period or a shutdown was requested.  The monitor issued the method: java.rmi.activation.Activatable.inactive() / or unregister(). RMI threw a RemoteException.

The server exits with a System.exit().

110 I  (L)

The String message for an ActivationException

No activity transpired for the requested period or a shutdown was requested.  The monitor issued the method: java.rmi.activation.Activatable.inactive() / or unregister().  RMI threw an ActivationException.

The server exits with a System.exit().

115 I  (P)

Tymeac Server, Logging NO longer in use from exception=

A log error occurred (the exception follows =).

Logging is no longer supported for this session.

120 I  (P)

Tymeac Server, Statistics NO longer in use from exception=

A statistics error occurred (the exception follows =).

Statistics are no longer supported for this session.

201 I  ( L, N)

Tymeac Server, activatable shut down may be stalled.

The Monitor had no activity for the time specified in the inactivate time. The server attempted to inactivate by running the shut down sequence but found that another task had previously started a normal shut down. Even after several attempts to continue, the Monitor is still waiting for the original shut down, stage 1, to complete.

When using shut down stage 1 user exits, one of these Classes may hang, thereby stalling the shut down.

Manual intervention may be required.

202 I  ( L, N)

Tymeac Server, activatable shut down stage 2 may be stalled.

The Monitor had no activity for the time specified in the inactivate time. The server attempted to inactivate by running the shut down sequence but found that another task had previously started a normal shut down. Even after several attempts to continue, the Monitor is still waiting for the original shut down, stage 2, to complete.

When using shut down stage 2 user exits, one of these Classes may hang, thereby stalling the shut down.

Manual intervention may be required.

901 S  (L, N)

Tymeac Server, Unexpected Shut Down from Monitor uncaught exception=

The Monitor threw an unhandled exception (the exception is after the =). A stack trace follows.

Tymeac cannot run without the Monitor. Therefore, the exception handler tries to shut down the system three times (normal shut down shutRequest(false) with a delay in between each request.) The exception handler then forces shut down with shutRequest(true).
