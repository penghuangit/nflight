RMI Registry Alternatives

Tymeac uses the Remote Method Invocation facility of Java. We discuss several alternative ways to structure your systems using this facility.

Preface

Tymeac was envisioned to be an accompaniment for you applications. That is, to reside with them on the same machine.

However, there is no limitation on how one may use Tymeac. Here, there and any step along the way. Tymeac is simply a single affinity service that multi-threads a request.

Restrictions

1.  For the RMI Registry, there is one major restriction [depending on you operating system] the CLASSPATH must point to the top level directory ("your directory"\Classes as <home>) in which the stubs reside. For Tymeac, the TymeacStandardImpl_Stub and TymeacActivationImpl_Stub reside in the <home>\com.tymeac.base directory. See also the RMI background document. This is not applicable when downloading via HTTP. In this case you need to specify the java.rmi.server.codebase property, where the server's classes will be made available, as <home>.

2.  The non-activatable Tymeac Server uses the String variables, TyVariables, for binding, (tymeac_name), at start up. Therefore, the non-activatable Tymeac Server must have access to this Class either in the directory, (<home>\com.tymeac.common).

3.  The Tymeac Clients, GUI and command line Classes, need to find the Tymeac Server. Both the non-activatable and activatable Systems use the String variables, TyVariables, for lookup() and an alternate procedure. Therefore, these Clients must have access to this Class either in the directory (<home>\com.tymeac.client[.jframe]).

4. Your user Processing Application Classes need to accessabiltiy by the Tymeac Server. Therefore, these classes must be in a directory under the <home> directory, (<home>\com.tymeac.serveruser).

Alternatives

You may have several Tymeac Client Spaces* using a single Tymeac Server. Please note the restriction on tymeac_name when using an activatable remote object, TyVariables.

Two TyVariable Classes with String variables:

1.  tymeac_name = "Tymeac_Red",
url_name = "rmi:///"

2.  tymeac_name = "Tymeac_Red",
url_name = "rmi://your-location.org/" or "rmi://192.168.42.1/"

Note the three slashes (///) for a local URL.  This is not a typo. That third slash is a trailing slash. In order to set this to a remote name, you put that remote name between the second and third slashes. Such as:
rmi://your-company.org/  or  rmi://192.168.42.1/

TyVariable (1) must go into the directory for use by the TymeacServer. TyVariable (2) may go into each Space* for each Tymeac Client.

For User Clients, you are free to use any home grown Class that better suits your needs.

You may have several Tymeac Client Spaces* using multiple Tymeac Servers.

Multiple TyVariable Classes with String variables:

1.  tymeac_name = "Tymeac_Red", url_name = "rmi:///"

2.  tymeac_name = "Tymeac_Green", url_name = "rmi:///"

3.  tymeac_name = "Tymeac_Blue", url_name = "rmi:///"

TyVariable (1, 2, 3) must go into the directory for each TymeacServer.   TyVariable (1, 2, 3) go into each Space* for use by each Tymeac Client.

For User Clients, you are free to use any home grown Class that better suits your needs.

 We supply an RMI communication Class, TySvrComm. This is a sample of how you may communicate with the Tymeac Server.  The Demonstration System uses this Class.

There are no restrictions for this Class. Please note that the Class uses the above TyVariable Class.

* A space may be anything from a Java name space within a single machine to an address space on another machine at another location.
